# الأمان والسلامة

تم تصميم Rosetta ليكون آمناً في البيئات المعادية — حيث قد تأتي بيانات اللغة من مصادر غير موثوقة، وحيث يمكن لأسماء الملفات المصممة خصيصاً تجاوز حدود الدليل، وحيث يمكن أن تحتوي مخرجات LLM على أي شيء.

## نموذج التهديد

| التهديد | ناقل الهجوم | التخفيف |
|--------|--------------|-----------|
| **Prototype pollution** | مفاتيح JSON مصممة خصيصاً (`__proto__`، `constructor`) | تُرفض وقت التحليل |
| **Path traversal** | رموز لغة مثل `../../etc/passwd` | يتم التحقق من صحة كتابة الملفات في الأدلة المكونة |
| **Code block corruption** | يقوم LLM بالترجمة داخل حدود التعليمات البرمجية | الحماية باستخدام علامات Unicode sentinel |
| **Hallucinated keys** | يُرجع LLM مفاتيح لم يتم إرسالها | التحقق من صحة الاستجابة — تتم كتابة المفاتيح المقبولة فقط |
| **Runaway token spend** | حلقات إعادة محاولة لا نهائية | مقيدة بميزانية عبر `maxRetries` |

## حماية Prototype Pollution

يتم التحقق من صحة جميع مفاتيح اللغة مقابل قائمة حظر قبل معالجتها:

- `__proto__`
- `constructor`
- `prototype`

يتم رفض أي مفتاح يطابق هذه الأنماط مع إرجاع خطأ. يمنع هذا المهاجمين من استخدام ملفات لغة مصممة خصيصاً لتعديل JavaScript object prototypes.

## احتواء المسار (Path Containment)

عند كتابة ملفات اللغة، يتحقق Rosetta من أن مسار المخرجات يبقى ضمن الأدلة المكونة (`localesDir`، `contentDir`). يتم تعقيم رموز اللغة — لا يمكن لرمز مثل `../../secrets` الكتابة خارج الدليل المتوقع.

## حماية الكتل (Block Protection)

أثناء ترجمة محتوى Markdown، يتم استبدال العناصر المهيكلة بعناصر نائبة من نوع Unicode sentinel قبل إرسال النص إلى LLM:

1. **Code blocks** (المحاطة والمضمنة) → sentinel
2. **Hugo shortcodes** (`{{< >}}`، `{{% %}}`) → sentinel  
3. **Raw HTML** → sentinel
4. **Interpolation variables** (`{{ .Count }}`) → sentinel

بعد الترجمة، يتم استبدال علامات sentinel بالمحتوى الأصلي. لا يرى LLM أبداً كتل التعليمات البرمجية أو الأكواد القصيرة أو HTML — وبالتالي لا يمكنه إتلافها.

## التحقق من صحة الاستجابة

عندما يُرجع LLM استجابة JSON، يتحقق Rosetta مما يلي:
- ظهور المفاتيح التي تم إرسالها في الدفعة فقط في الاستجابة
- عدم إدخال أي مفاتيح إضافية
- إمكانية تحليل الاستجابة كـ JSON صالح

يتم إسقاط المفاتيح المهلوسة (Hallucinated keys) بصمت. يمنع هذا مخرجات LLM من إدخال ترجمات غير متوقعة في ملفات اللغة الخاصة بك.

## بوابة الجودة (Quality Gate)

يتم التحقق من صحة كل ترجمة من خلال خمسة فحوصات حتمية قبل كتابتها على القرص. راجع [بوابة الجودة (Quality Gate)](/docs/concepts/quality-gate) للحصول على التفاصيل.

## التراجع الأسي (Exponential Backoff)

تستخدم استدعاءات API التراجع الأسي (exponential backoff) مع التذبذب (jitter) عند استلام استجابات 429 (حد المعدل) و 5xx (خطأ في الخادم). تمنع ثلاث محاولات إعادة مع تأخير متزايد الضغط الزائد على API أثناء فترات الانقطاع.

## مهلة الطلب

يحتوي كل طلب API على مهلة قدرها 30 ثانية عبر `AbortController`. يمنع هذا عملية المزامنة من التعليق إلى أجل غير مسمى بسبب اتصال ميت.

## وضع التراجع (Fallback Mode)

عندما تكون API غير متوفرة، يكتب `--fallback` عناصر نائبة مسبوقة بـ `[EN]` بدلاً من الترجمات الحقيقية:

```bash
npx i18n-rosetta sync --fallback
```

```json
{
  "hero.title": "[EN] Welcome to our platform"
}
```

يتم اكتشاف هذه العناصر النائبة تلقائياً وإعادة ترجمتها في المزامنة التالية باستخدام مفتاح API صالح. لا يتم التعامل معها أبداً على أنها "مترجمة" — سيقوم `audit` بوضع علامة عليها.

## الاختبار

يتم التحقق من خصائص الأمان بواسطة مجموعة الاختبارات المعادية (adversarial test suite):

```bash
npm run test:redteam    # prototype pollution, path traversal, encoding attacks
```
