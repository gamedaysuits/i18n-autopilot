# استكشاف الأخطاء وإصلاحها

المشكلات الشائعة وحلولها لـ i18n-rosetta.

## واجهة برمجة التطبيقات (API) والمصادقة

### "OPENROUTER_API_KEY not found"

يتطلب Rosetta مفتاح API لترجمة النماذج اللغوية الكبيرة (LLM). قم بتعيينه كمتغير بيئة:

```bash
export OPENROUTER_API_KEY="sk-or-v1-..."
```

أو في ملف `.env` (إذا كان مشروعك يقوم بتحميل ملفات `.env`):

```
OPENROUTER_API_KEY=sk-or-v1-...
```

:::tip
إذا كان لديك مفتاح API لـ Google Translate فقط، سيكتشف rosetta ذلك تلقائيًا ويستخدم Google Translate كطريقة افتراضية. لا حاجة لتغيير الإعدادات.
:::

### "401 Unauthorized" من OpenRouter

مفتاح API الخاص بك غير صالح أو منتهي الصلاحية. تحقق منه عبر [openrouter.ai/keys](https://openrouter.ai/keys).

### "429 Too Many Requests" / تقييد معدل الطلبات (Rate Limiting)

يتعامل Rosetta مع حدود معدل الطلبات داخليًا باستخدام التراجع الأسي (exponential backoff). إذا كنت تصل باستمرار إلى حدود معدل الطلبات:

1. **تقليل حجم الدفعة (batch size)** في إعداداتك:
   ```json
   { "batchSize": 15 }
   ```
2. **استخدام نموذج بحدود أعلى لمعدل الطلبات** (على سبيل المثال، `google/gemini-3.5-flash` لديه حدود سخية)
3. **استخدام طريقة أرخص/أسرع** للأزواج اللغوية ذات الحجم الكبير — لا يوجد لدى Google Translate حدود لمعدل الطلبات:
   ```json
   { "pairs": { "en:it": { "method": "google-translate" } } }
   ```

### لم يتم العثور على النموذج (Model Not Found) / أخطاء 404

يقوم مزودو LLM المباشرون (`openai`، `anthropic`، `gemini`) بالتحقق من صحة سلسلة النموذج (model string) عند الاستخدام الأول. إذا رأيت تحذيرًا:

**"looks like an OpenRouter path"** — أنت تستخدم نموذجًا بتنسيق OpenRouter (`google/gemini-3.5-flash`) مع مزود مباشر. يستخدم المزودون المباشرون أسماء النماذج المجردة:

```diff
- { "method": "gemini", "model": "google/gemini-3.5-flash" }
+ { "method": "gemini", "model": "gemini-2.5-flash" }
```

أو قم بالتبديل إلى طريقة `llm` لاستخدام OpenRouter:
```json
{ "method": "llm", "model": "google/gemini-3.5-flash" }
```

**"is an Anthropic/OpenAI/Gemini model"** — أنت ترسل نموذجًا إلى المزود الخطأ:

```diff
- { "method": "gemini", "model": "claude-sonnet-4-6" }
+ { "method": "anthropic", "model": "claude-sonnet-4-6" }
```

**"not found in available models"** — قد يكون النموذج مهملًا (deprecated) أو مكتوبًا بشكل خاطئ. يجلب Rosetta قائمة النماذج المباشرة للمزود ويقترح بدائل. تحقق من وثائق المزود لمعرفة أسماء النماذج الحالية.

:::tip إهمال النماذج أمر وارد
يقوم المزودون بإيقاف أسماء النماذج بانتظام. إذا فشلت الترجمات فجأة بعد تحديث من المزود، فتحقق من مخرجات `[WARN]` — ستعرض لك البدائل الحالية.
:::

## جودة الترجمة

### الترجمات تكرر لغة المصدر

تلتقط بوابة الجودة (quality gate) هذا الأمر. إذا كانت الترجمة مطابقة للمصدر الإنجليزي، يتم رفضها وإعادة المحاولة. إذا استمرت المشكلة:

1. **التحقق من النموذج** — أداء بعض النماذج ضعيف في أزواج لغوية معينة
2. **إضافة تعليمات الأسلوب (register instructions)** — أخبر النموذج باللغة التي يجب إنتاجها:
   ```json
   {
     "languages": {
       "ja": { "name": "Japanese", "register": "Polite/formal Japanese" }
     }
   }
   ```
3. **تجربة نموذج مختلف** — قم بالتبديل من `gpt-4o-mini` إلى `gpt-4o` أو `google/gemini-2.5-pro`

### مخرجات بنظام كتابة خاطئ (مثل نص لاتيني لليابانية)

يلتقط فحص التوافق مع نظام الكتابة في بوابة الجودة معظم الحالات. إذا استمرت المشكلة:

- تحقق من صحة رمز اللغة (locale code) (`ja`، وليس `jp`)
- أضف تعليمات صريحة لنظام الكتابة في حقل `register`:
  ```json
  { "register": "Japanese using hiragana, katakana, and kanji" }
  ```

### أنماط الهلوسة في المخرجات

يتم التقاط أنماط الترايغرام (trigram) المتكررة (مثل "مرحبا مرحبا مرحبا") بواسطة كاشف حلقات الهلوسة. إذا كانت المخرجات مشوهة ولكنها تجاوزت الكاشف:

1. **تقليل حجم الدفعة** — الدفعات الأصغر تنتج مخرجات أكثر تركيزًا
2. **استخدام نموذج أقوى** — النماذج الأكبر تهلوس بشكل أقل في أنظمة الكتابة غير اللاتينية
3. **إضافة بيانات توجيهية (coaching data)** — مصطلحات القاموس تعمل كمرتكز للترجمة

## مشكلات الملفات والتنسيق

### "No locale files found"

يكتشف Rosetta ملفات اللغة تلقائيًا. إذا لم يتمكن من العثور عليها:

1. **التحقق من `localesDir`** — يجب أن يشير إلى الدليل الذي يحتوي على ملفات اللغة:
   ```json
   { "localesDir": "./locales" }
   ```
2. **التحقق من تسمية الملفات** — يجب تسمية الملفات برمز اللغة: `en.json`، `fr.json`، إلخ.
3. **التحقق من التنسيق** — التنسيقات المدعومة: JSON، nested JSON، YAML، TOML

### تعارضات ملف القفل (Lock file)

إذا أصبح `.i18n-rosetta.lock` في حالة سيئة:

```bash
# Reset the lock file (next sync will retranslate everything)
rm .i18n-rosetta.lock
npx i18n-rosetta sync
```

:::warning
حذف ملف القفل يعني أن المزامنة التالية ستعيد ترجمة جميع المفاتيح، وليس فقط المفاتيح التي تم تغييرها. هذا له آثار على تكلفة API للمشاريع الكبيرة.
:::

### إعادة ترجمة مفاتيح محددة

إذا كانت هناك ترجمات فردية خاطئة وأردت إجبار النظام على إعادة ترجمتها دون حذف ملف القفل:

```bash
# Re-translate a single key
npx i18n-rosetta sync --force-keys "hero.title"

# Re-translate multiple keys
npx i18n-rosetta sync --force-keys "nav.home,nav.about,footer.copyright"
```

تتجاوز العلامة `--force-keys` فحص تجزئة (hash) ملف القفل لتلك المفاتيح المحددة، مما يجبر النظام على إعادة الترجمة دون التأثير على أي مفاتيح أخرى.

### ترجمة المحتوى تفسد كتل التعليمات البرمجية

لا ينبغي أن يحدث هذا — حيث يتم حماية كتل التعليمات البرمجية (code blocks) قبل الترجمة. إذا حدث ذلك:

1. تحقق من أن كتلة التعليمات البرمجية تستخدم التنسيق القياسي (ثلاث علامات اقتباس مائلة)
2. تحقق من وجود كتل تعليمات برمجية غير مغلقة في مصدر Markdown
3. قم بفتح مشكلة (File an issue) — هذا خطأ برمجي (bug) في نظام الحماية (sentinel shielding system)

## مشكلات واجهة سطر الأوامر (CLI)

### `--watch` لا يكتشف التغييرات

تستخدم مراقبة الملفات وظيفة `fs.watch` الأصلية في Node.js. المشكلات المعروفة:

- **محركات أقراص الشبكة** — لا تعمل `fs.watch` بشكل موثوق على مسارات NFS/SMB
- **وحدات تخزين Docker** — استخدم وضع الاستقصاء (polling mode) أو قم بتشغيل rosetta داخل الحاوية
- **الأدلة الكبيرة** — يراقب النظام `localesDir` بشكل متكرر (recursively)؛ قد تتجاوز الأشجار العميقة جدًا حدود نظام التشغيل

### `npx` يقوم بتشغيل إصدار قديم

```bash
# Clear the npx cache
npx --yes i18n-rosetta@latest sync
```

أو قم بتثبيته بشكل عام (globally):

```bash
npm install -g i18n-rosetta
i18n-rosetta sync
```

## الأداء

### المزامنة بطيئة للعديد من اللغات

يترجم Rosetta جميع اللغات بالتوازي افتراضيًا. إذا كانت المزامنة لا تزال بطيئة:

1. **استخدام Google Translate للأزواج اللغوية ذات الحجم الكبير** — فهو أسرع بـ 10 إلى 50 ضعفًا من ترجمة LLM
2. **زيادة حجم الدفعة** (الافتراضي هو 80):
   ```json
   { "batchSize": 120 }
   ```
3. **ضبط التزامن (concurrency)** — التوازي الافتراضي للغات JSON هو 50 وللمحتوى هو 12. إذا كان مزود API الخاص بك يدعم حدودًا أعلى لمعدل الطلبات:
   ```bash
   npx i18n-rosetta sync --json-concurrency 80 --content-concurrency 20
   ```
4. **استخدام نموذج سريع** — `gpt-4o-mini` أسرع بكثير من `gpt-4o`

### ارتفاع تكاليف API

- **التحقق من أحجام الدفعات** — دفعات أكبر = استدعاءات API أقل = تكلفة أقل
- **استخدام ذاكرة الترجمة (Translation Memory)** — تعمل ذاكرة الترجمة (TM) افتراضيًا. قم بتشغيل `i18n-rosetta tm stats` للتحقق من عملها. إذا رأيت 0 إدخالات بعد عمليات مزامنة متعددة، فقد يكون هناك خطأ ما في أذونات دليل `.rosetta/` الخاص بك
- **استخدام التخزين المؤقت للمطالبات (prompt caching)** — يقسم Rosetta رسائل النظام/المستخدم لتحقيق إصابات التخزين المؤقت (cache hits) في نماذج Anthropic و Google
- **استخدام Google Translate للغات المستوى الثاني (Tier 2)** — راجع دليل [ترجمة 30 لغة](/docs/tutorials/translate-30-languages)

### ترجمات قديمة (Stale) بعد تبديل المزودين

إذا قمت بالتبديل من طريقة ترجمة إلى أخرى (على سبيل المثال، من `llm` إلى `deepl`)، فقد تستمر ذاكرة التخزين المؤقت لـ TM في تقديم ترجمات قديمة من الطريقة السابقة للمفاتيح التي لم يتغير نصها المصدر. يتضمن مفتاح التخزين المؤقت (cache key) اسم الطريقة، لذلك يتم التعامل مع معظم الحالات تلقائيًا. ولكن إذا قمت بتغيير `model` داخل نفس الطريقة:

```bash
# Force fresh translations for all keys
i18n-rosetta sync --no-tm

# Or clear the cache entirely and re-sync
i18n-rosetta tm clear --yes
i18n-rosetta sync
```

راجع [ذاكرة الترجمة](/docs/concepts/translation-memory) للحصول على تفاصيل حول تصميم مفتاح التخزين المؤقت.
