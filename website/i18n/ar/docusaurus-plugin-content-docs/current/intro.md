# i18n-rosetta

إطار عمل دولي (internationalization) قابل للتخصيص بالكامل. أمر واحد يترجم ملفات اللغات الخاصة بك. إعداد واحد يتحكم في كل طريقة، ونموذج، وزوج لغوي. وإذا لم تكن الطرق المدمجة كافية — يمكنك بناء طريقتك الخاصة، وإثبات فعاليتها، ثم نشرها.

```bash
npx i18n-rosetta sync
```

يكتشف rosetta تلقائيًا ملفات اللغات الخاصة بك، وتنسيقها، واللغات المستهدفة. يقوم بترجمة ما هو مفقود، ويتخطى ما تم إنجازه، ويتحقق من صحة كل نتيجة، ويكتب مخرجات نظيفة. هذه هي نقطة البداية.

---

## لماذا لا تبرمجها بنفسك؟

يمكنك كتابة حلقة تكرار (loop) سريعة تستدعي Google Translate لكل مفتاح. يفعل معظم المطورين ذلك — يستغرق الأمر حوالي 30 سطرًا. ولكن هنا تكمن المشكلة:

- **عدم اكتشاف التغييرات.** إذا قمت بتحديث نص إنجليزي — تظل الترجمة قديمة إلى الأبد. يتتبع rosetta كل قيمة مصدر باستخدام تجزئات SHA-256 ويعيد ترجمة ما تغير فقط.
- **عدم وجود تجميع (Batching).** استدعاء واجهة برمجة تطبيقات (API) واحد لكل مفتاح يعني 200 مفتاح = 200 رحلة ذهاب وإياب. يقوم rosetta بالتجميع بذكاء (قابل للإعداد، الافتراضي 30 مفتاحًا/دفعة لـ LLM، و128 لـ Google).
- **عدم وجود بوابة جودة.** الترجمة الآلية تهلوس، أو تعيد النص المصدر كما هو، أو تخرج نصًا بأبجدية خاطئة. يتحقق rosetta من صحة كل ترجمة قبل كتابتها — حيث يتم اكتشاف ورفض الأبجدية الخاطئة، والتضخم في الطول، وتكرار المصدر.
- **عدم الوعي بالتنسيق.** هل هو مبرمج ثابتًا (Hardcoded) لـ JSON؟ يتعامل rosetta مع JSON و TOML و YAML و Hugo Markdown (البيانات الوصفية + النص) مع الاكتشاف التلقائي.
- **عدم التحكم في الطريقة.** يحصل كل زوج لغوي على نفس الطريقة. يتيح لك rosetta استخدام Google Translate للغة الفرنسية، و LLM للغة اليابانية، ومسار عمل (pipeline) مخصص مستضاف مجتمعيًا للغة Cree — في نفس ملف الإعدادات.

rosetta هو إصدار الإنتاج (production version) من ذلك النص البرمجي.

## ما الذي يجعله مختلفًا

### كل طريقة هي مكون إضافي (Plugin)

طريقة الترجمة **قابلة للإعداد لكل زوج لغوي**. يمكنك المزج بين Google Translate و LLMs والمطالبات الموجهة (coached prompts) وواجهات برمجة التطبيقات (APIs) المخصصة في نفس المشروع:

```json title="i18n-rosetta.config.json"
{
  "version": 3,
  "pairs": {
    "en:fr": { "method": "google-translate" },
    "en:ja": { "method": "llm", "model": "google/gemini-2.5-pro" },
    "en:crk": { "methodPlugin": "crk-coached-v1" }
  }
}
```

تحصل الفرنسية على Google Translate (سريع ورخيص). وتحصل اليابانية على LLM متميز (دقيق). وتحصل Plains Cree على مكون إضافي موجه بقواعد نحوية وقواميس وتحقق صرفي. نفس الأمر `sync`. نفس بوابة الجودة. نفس واجهة سطر الأوامر (CLI).

### أثبت ذلك

هل تعتقد أن طريقتك يمكنها ترجمة الإنجليزية إلى الإسبانية؟ أو التركية إلى الأذربيجانية؟ أو الإنجليزية إلى Cree؟

**أثبت ذلك.** تقوم [أداة التقييم (eval harness)](/docs/eval/harness) المرافقة بقياس أداء أي طريقة ترجمة بنتائج قابلة للتكرار وموثقة ببصمة. وتتتبع [لوحة المتصدرين](/leaderboard) كل إرسال.

تشترك أداة التقييم وواجهة سطر الأوامر (CLI) الخاصة بالإنتاج في نفس واجهة المكونات الإضافية. الطريقة التي تسجل درجات جيدة في التقييم يمكن استخدامها في الإنتاج — إذا وافق المجتمع الذي تخدم لغته على ذلك. بالنسبة للغات الأصلية واللغات ذات الموارد المحدودة، هذه الموافقة مهمة. راجع [سيادة البيانات (Data Sovereignty)](/docs/guides/data-sovereignty).

```bash
# Benchmark your method (in the eval harness repo)
cd gds-mt-eval-harness
python eval/baseline_experiment.py --dataset data/edtekla-dev-v1.json --submit

# Use it locally
npx i18n-rosetta sync
```

نفس المكون الإضافي. قم بالتوصيل والاختبار.

### مجموعة الأدوات الكاملة

لا يقتصر rosetta على `sync` فحسب. بل هو مسار عمل (pipeline) كامل للتدويل (i18n):

| الأمر (Command) | وظيفته |
|---------|-------------|
| `sync` | ترجمة المفاتيح المفقودة، والقديمة، والاحتياطية (fallback) |
| `watch` | المزامنة التلقائية عند تغير الملف المصدر |
| `lint` | فحص الكود المصدري بحثًا عن النصوص المبرمجة ثابتًا (hardcoded) |
| `wrap` | التغليف التلقائي للنصوص المبرمجة ثابتًا في استدعاءات `t()` |
| `audit` | سرد جميع القيم الاحتياطية `[EN]` غير المترجمة |
| `integrity` | اكتشاف تلف العناصر النائبة (placeholders) ومشاكل الترميز |
| `seo` | إنشاء علامات hreflang، وخرائط الموقع (sitemaps)، و JSON-LD |
| `status` | عرض إعدادات الزوج اللغوي، والمكونات الإضافية، ودرجات قياس الأداء |
| `provenance` | تدقيق تراخيص موارد الترجمة |
| `plugin` | تثبيت، وإزالة، وسرد المكونات الإضافية للطرق |

ثلاثة من هذه الأوامر — `lint`، `sync`، `audit` — تشكل مسار عمل للتكامل المستمر (CI pipeline) يكتشف النصوص المبرمجة ثابتًا، ويترجمها، ويوقف عملية البناء (build) إذا كانت أي لغة غير مكتملة.

## الساحة (The Arena)

[لوحة متصدري الطرق](/leaderboard) هي لوحة النتائج. يتم توثيق كل إرسال ببصمة مرتبطة بتأكيد (commit) في Git، وتعيين إصدار لمجموعة بيانات محددة، وتقييمه بواسطة نفس الأداة. يمكن لأي شخص الإرسال.

**ما الذي يمكنك إثباته؟** تأخذ أداة التقييم تنسيق JSON. وتأخذ المكونات الإضافية تنسيق JSON. يمكن اختبار أي طريقة تنتج JSON:

| النهج | مثال |
|----------|---------|
| **LLM موجه (Coached LLM)** | إدخال القواعد النحوية والقواميس في مطالبة (prompt) نموذج متقدم |
| **نموذج مضبوط دقيقًا (Fine-tuned)** | تدريب نموذج مفتوح على نصوص متوازية — ولكن ليس على بيانات التقييم |
| **مسار عمل ببوابة FST** | يقوم LLM بالإنشاء ← يتحقق محول الحالة المحدودة (FST) من الصرف ← إعادة المحاولة |
| **نماذج متسلسلة (Chained)** | النموذج A يصيغ ← النموذج B يحرر لاحقًا ← النموذج C يقيّم |
| **قاموس + LLM** | فرض مصطلحات معروفة من قاموس، وترك LLM يتعامل مع الباقي |
| **تطوري (Evolutionary)** | إنشاء مرشحين، وتقييمهم، وتعديل الأفضل، ثم التكرار |
| **ترجمة جزئية** | ترجمة عينة يدويًا، وإثبات تطابق LLM الخاص بك، ثم الترجمة التلقائية للباقي |

قم بالضبط الدقيق للنماذج. انشر خوارزميات تطورية. اختبر إجابات الطلاب في امتحانات اللغة. ابنِ جداول بحث (lookup tables). اربط ثلاثة نماذج معًا. طالما أن طريقتك تنتج JSON، فإن أداة التقييم ستقيمها وسيقوم إطار العمل بتشغيلها.

:::danger القاعدة الوحيدة
**لا تتدرب على بيانات التقييم.** سيتم استبعاد الطرق التي تعرضت لمجموعة بيانات قياس الأداء. قم بالضبط الدقيق على أي شيء تريده. ولكن ليس على مجموعة الاختبار.
:::

هذه دعوة مفتوحة. إذا كنت تعمل مع لغة ذات موارد محدودة — كباحث، أو عضو في المجتمع، أو طالب، أو مجرد شخص مهتم — فقم ببناء طريقة، وشغل أداة التقييم، واحصل على أعلى درجة. المشكلة لم تُحل بعد. والبنية التحتية موجودة هنا.
