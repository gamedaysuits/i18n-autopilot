Step-by-step setup po para sa i18n-rosetta gamit ang mga popular frameworks.

## Hugo (TOML / YAML / Markdown)

### Project Structure

Gumagamit ang Hugo ng `i18n/` para sa string translations at `content/` para sa page content:

```
my-hugo-site/
├── i18n/
│   ├── en.toml             ← source of truth
│   ├── fr.toml
│   └── ja.toml
├── content/
│   ├── posts/
│   │   ├── hello.md        ← source (English)
│   │   ├── hello.fr.md
│   │   └── hello.ja.md
│   └── about.md
└── .env.local
```

### Setup

```bash
npm install --save-dev i18n-rosetta
```

```json title="i18n-rosetta.config.json"
{
  "version": 3,
  "inputLocale": "en",
  "localesDir": "./i18n",
  "contentDir": "./content",
  "format": "auto",
  "languages": ["fr", "de", "ja", "es", "ko", "zh"]
}
```

```bash
i18n-rosetta sync           # sync i18n string files + content files
i18n-rosetta sync --dry     # preview changes without writing
```

### Content Translation Details

**Front matter**: Supported po ang parehong YAML (`---`) at TOML (`+++`) delimiters. Tina-translate nito ang `title`, `description`, `summary`, `subtitle`, `caption`, at `linkTitle` by default. Naka-preserve naman ang lahat ng iba pang fields (date, draft, tags, weight, slug, etc.). Pwede niyo itong i-customize gamit ang `translatableFields` sa inyong config.

**Block protection**: Awtomatikong naka-shield ang mga code blocks, Hugo shortcodes, inline code, at raw HTML gamit ang Unicode sentinel placeholders. Magpa-pass through lang po ang mga ito nang hindi nagagalaw.

**Filename convention**: Sinusunod nito ang translation-by-filename pattern ng Hugo:
- `my-post.md` → `my-post.fr.md`
- `my-post.en.md` → `my-post.fr.md` (tinatanggal ang source suffix)

**Skip existing**: Hindi po ino-overwrite ang mga existing translated files. I-delete ang isang target file para i-force ang re-translation.

### Plural Forms

Supported ng TOML at YAML locales ang CLDR plural forms:

```toml
[items]
one = "{{ .Count }} item"
other = "{{ .Count }} items"
```

Internally represented ito bilang `items.one` at `items.other` para sa diffing, tapos ire-re-serialize sa tamang sectioned format kapag nag-write na.

## next-intl (JSON)

### Project Structure

```
my-app/
├── messages/
│   └── en.json        ← source of truth
├── src/
│   ├── i18n/
│   │   ├── routing.ts
│   │   └── request.ts
│   └── middleware.ts
└── .env.local
```

### Setup

```bash
npm install --save-dev i18n-rosetta
```

```json title="i18n-rosetta.config.json"
{
  "version": 3,
  "inputLocale": "en",
  "localesDir": "./messages",
  "languages": ["fr", "de", "ja", "es", "ko", "zh", "pt", "ar"]
}
```

```bash
npx i18n-rosetta sync
```

Gagawa ito ng `messages/fr.json`, `messages/ja.json`, etc. — fully translated, at naka-preserve ang inyong nested key structure. Awtomatiko po itong ipi-pick up ng next-intl.

### Development Workflow

```json title="package.json"
{
  "scripts": {
    "dev": "i18n-rosetta watch & next dev",
    "i18n:sync": "i18n-rosetta sync",
    "i18n:audit": "i18n-rosetta audit"
  }
}
```

## react-i18next (JSON)

### Flat File Structure (recommended)

```
locales/
├── en.json
├── fr.json
└── ja.json
```

```json title="i18n-rosetta.config.json"
{
  "version": 3,
  "inputLocale": "en",
  "localesDir": "./locales",
  "languages": ["fr", "de", "ja"]
}
```

```bash
npx i18n-rosetta sync
```

### Nested Directory Structure

Kung ginagamit niyo ang `{locale}/{namespace}.json` structure, gumawa po ng sync script para mag-flatten → translate → unflatten. Tingnan ang [react-i18next docs](https://react.i18next.com/) para sa details.
