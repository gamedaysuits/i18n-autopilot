# Rosettaの比較

i18n-rosettaは、ほとんどのローカリゼーションツールとは異なるカテゴリに属しています。ここでは、率直な比較をご紹介します。

## 全体像

ほとんどのローカリゼーションツールは、以下の3つのカテゴリのいずれかに分類されます。

| カテゴリ | 例 | モデル |
|----------|----------|-------|
| **クラウドTMSプラットフォーム** | Crowdin, Phrase, Locize, Tolgee | SaaSダッシュボード + 人間による翻訳 + 月額サブスクリプション |
| **キー抽出ツール** | i18next-scanner, FormatJS CLI | ソースコードをスキャンして翻訳関数の呼び出しを抽出 |
| **CLI翻訳エンジン** | **i18n-rosetta** | プロジェクト内で実行し、ファイルを直接翻訳。クラウドアカウント不要 |

Rosettaは**CLI翻訳エンジン**です。設定可能なバックエンド（LLM、Google翻訳、カスタムプラグイン）を使用して、ロケールファイルを直接翻訳します。クラウドダッシュボードや、人間による翻訳ワークフロー、月額料金は一切ありません。

## Rosettaを使うべきケース

**Rosettaは以下のような場合に適しています：**

- 機械翻訳を別のワークフローとしてではなく、ビルドパイプラインに組み込みたい場合
- 言語ごとにメソッドを制御したい場合（一部の言語にはLLM、他の言語にはGoogle翻訳、残りの言語にはカスタムプラグインを使用するなど）
- APIが対応していない言語（先住民言語、消滅危機言語、人工言語など）に翻訳する場合
- 決定論的な文字スクリプト出力が必要な場合（クリー文字、クリンゴン語のpIqaD、テングワールなど）
- ベンダーロックインやクラウドへの依存を完全になくしたい場合
- 人間による翻訳ワークフローを必要としない、個人開発者や小規模チームである場合

**クラウドTMSは以下のような場合により適しています：**

- プロの翻訳者がすべての文字列をレビューする場合
- プロジェクト間で翻訳メモリや用語集の管理が必要な場合
- インコンテキストの視覚的な編集（UI内で翻訳をプレビューすること）が必要な場合
- ロールベースのアクセス制御を必要とする大規模なチームである場合
- 50以上のファイルフォーマットへの対応が必要な場合

## Rosettaにしかできないこと

### 1. カスタムレジスター

すべての言語ペアにおいて、LLMに対して文化的に適切なトーン（文体）の指示が与えられます：

```json
{
  "de": {
    "register": "Standard professional register. Use Sie-form for formal address."
  },
  "tl": {
    "register": "Educated Manila Taglish. Use Tagalog as the primary language but keep technical terms in English."
  },
  "tlh": {
    "register": "Warrior's honor. OVS grammar. Use Marc Okrand vocabulary."
  }
}
```

47種類の言語レジスターを標準搭載し、さらにプロジェクトごとにカスタムレジスターを定義できるツールは他にありません。

### 2. 決定論的な文字スクリプトコンバーター

Rosettaには、翻訳後のフックとして実行される5つの組み込みスクリプトコンバーターが搭載されており、LLMは必要ありません：

| ロケール | 変換 | 例 |
|--------|-----------|---------|
| `crk` | SRO → クリー文字 | `nêhiyawêwin` → `ᓀᐦᐃᔭᐍᐏᐣ` |
| `sr` | ラテン文字 → キリル文字 | `Beograd` → `Београд` |
| `tlh` | ローマ字表記 → pIqaD | `tlhIngan Hol` → (pIqaDのグリフ) |
| `x-elvish-s` | ラテン文字 → テングワール | シンダリン → テングワール (ベレリアンド・モード) |
| `x-kryptonian` | ラテン文字 → クリプトン文字 | 暗号置換（フォントが必要） |

これらは純粋なルックアップテーブルによるコンバーターであり、決定論的で監査可能であり、LLMのハルシネーション（幻覚）のリスクはゼロです。

### 3. コンテンツ認識シールド

Markdownやリッチコンテンツを翻訳する際、Rosettaは以下の要素を保護します：

- フェンスコードブロック (` ``` `)
- インラインコード (`` ` ` ``)
- Hugoショートコード (`{{</* */>}}`, `{{%/* */%}}`)
- 補間変数 (`{{ .Count }}`, `{name}`, `{{t('key')}}`)
- 生のHTMLブロック

これらは翻訳前にUnicodeのセンチネルトークンに置き換えられ、翻訳後に復元されます。LLMがコードやショートコード、変数を参照することは決してありません。

### 4. コーチングメソッドプラグイン

APIが対応していない言語については、コーチングされた翻訳メソッドを構築することができます：

1. 言語的なコーチングデータ（文法規則、語彙、例文）を作成する
2. プラグインとしてバンドルする
3. [評価ハーネス](https://github.com/gamedaysuits/gds-mt-eval-harness)を使用して、参照翻訳に対するベンチマークテストを行う
4. `i18n-rosetta plugin install`を使用してプロジェクトにインストールする

これはRosettaが平原クリー語を処理する方法であり、まだ存在しない言語を含め、あらゆる言語を処理できる方法でもあります。

## 結論

RosettaはCrowdinの代替品ではありません。異なるワークフローのための異なるツールです。人間による翻訳が必要な場合は、TMSを使用してください。1つのコマンドでファイルを翻訳し、メソッド、モデル、レジスターを言語ごとに制御できるCLIが必要な場合は、Rosettaを使用してください。
