# データ主権

先住民言語の機械翻訳には、フランス語や日本語には存在しない疑問が生じます。トレーニングデータの所有者は誰か？言語モデルの話し方を誰が管理するのか？翻訳が公開に足る品質かどうかを誰が判断するのか？

**その答えは、常にコミュニティです。**

rosettaはこれをサポートするために構築されています。`api` メソッドは、すべての言語リソースをサーバー側に保持し、コミュニティの管理下に置きます。プラグインシステムは、メソッドをツールから分離します。しかし、ツールが倫理を強制することはできません。このページでは、従うべき原則について説明します。

---

## OCAP® 原則

**OCAP**（Ownership: 所有、Control: 管理、Access: アクセス、Possession: 保持）は、[First Nations Information Governance Centre](https://fnigc.ca/ocap-training/)（FNIGC）によって開発された原則であり、ファースト・ネーションズ（カナダの先住民）のデータをどのように収集、保護、使用、共有すべきかを定めています。

| 原則 | 翻訳における意味 |
|-----------|------------------------------|
| **Ownership (所有)** | コミュニティは、辞書、文法書、対訳テキスト、コーチングファイル、およびそれらから生成されたすべての翻訳など、自身の言語データを所有します。 |
| **Control (管理)** | コミュニティは、言語データがどのように使用されるか、誰がアクセスできるか、どの翻訳メソッドが許容されるかを管理します。 |
| **Access (アクセス)** | コミュニティのメンバーは、保存場所に関係なく、自身の言語リソースにアクセスし、管理する権利を持ちます。 |
| **Possession (保持)** | 物理的なデータ（コーチングファイル、辞書、モデルの重み）は、サードパーティのクラウドではなく、コミュニティが管理するインフラストラクチャ上に存在する必要があります。 |

### OCAPの実際的な意味

- コミュニティの明示的な許可なしに、先住民言語の**翻訳を公開しないでください**。
- データ共有協定なしに、コミュニティが提供した言語データで**モデルをトレーニングしないでください**。
- ウェブサイト、ソーシャルメディア、または教育資料からコミュニティの言語リソースを**スクレイピングしないでください**。
- プロンプト、コーチングデータ、辞書がコミュニティ管理のサーバーに留まるように、**`api` メソッドを使用してください**。rosettaの `api` メソッドは「ダムパイプ（単なるデータの通り道）」であり、キーを送信して翻訳を受け取るだけです。すべての言語的な知的財産（IP）はサーバー側に保持されます。
- **出所の文書化** — [プラグインマニフェスト](/docs/reference/plugin-spec)の `provenance` フィールドには、使用されたすべてのリソース、そのライセンス、およびその出所を記載する必要があります。

:::warning OCAP®は登録商標です
OCAP®は、First Nations Information Governance Centreの登録商標です。これは特にカナダのファースト・ネーションズに適用されます。この原則はより広範な関連性を持ちますが、商標およびガバナンスの権限はFNIGCに帰属します。
:::

## CARE 原則

**先住民データガバナンスのためのCARE原則**は、FAIRデータ原則を補完するものとして、[Global Indigenous Data Alliance](https://www.gida-global.org/care)（GIDA）によって開発されました。FAIRは、データがFindable（見つけやすく）、Accessible（アクセス可能で）、Interoperable（相互運用可能で）、Reusable（再利用可能）であるべきだとしています。CAREは、それだけでは不十分であり、データガバナンスは先住民の権利も中心に据えなければならないとしています。

| 原則 | 適用 |
|-----------|------------|
| **Collective Benefit (集団の利益)** | 翻訳ツールは、まず言語コミュニティに利益をもたらすものでなければなりません。リーダーボードのスコアはメソッドを改善するための手段であり、コミュニティの言語から商業的価値を引き出すためのものではありません。 |
| **Authority to Control (管理する権限)** | コミュニティは、自身の言語データがどのように収集、使用、共有されるかを統治する権限を持ちます。リーダーボードのスコアが高いからといって、翻訳を公開する許可が得られるわけではありません。 |
| **Responsibility (責任)** | 先住民の言語データを扱う研究者や開発者は、関係を構築し、同意を得て、利益を共有する責任があります。 |
| **Ethics (倫理)** | 先住民の権利と幸福を最優先事項としなければなりません。翻訳メソッドは、コミュニティ「について」ではなく、コミュニティ「とともに」開発されるべきです。 |

## Te Mana Raraunga — マオリのデータ主権

**Te Mana Raraunga**は、[マオリ・データ主権ネットワーク](https://www.temanararaunga.maori.nz/)です。言語データを含むマオリのデータは、ワイタンギ条約の原則およびティカンガ・マオリ（マオリの慣習法）の対象となるタオンガ（宝物）であると主張しています。

主要な原則:

| 原則 | 意味 |
|-----------|---------|
| **Rangatiratanga** (権限) | マオリは、言語データを含む自身のデータに対して権限を行使する固有の権利を持ちます。 |
| **Whakapapa** (関係性) | データには起源とつながりがあります。言語データには、それを作成した人々の関係性と知識が含まれています。 |
| **Whanaungatanga** (義務) | マオリのデータを保持または処理する者は、そのデータの出所であるコミュニティに対して相互の義務を負います。 |
| **Kotahitanga** (集団の利益) | マオリのデータは、マオリの集団的利益のために使用されるべきです。 |
| **Manaakitanga** (互恵性) | マオリのデータの使用には、配慮、敬意、および互恵性が伴うべきです。 |
| **Kaitiakitanga** (後見) | データの保護者は、データを保護し、それが適切に使用されることを保証する義務を負います。 |

これらの原則は、te reo Māori（マオリ語）およびマオリ語データを含むあらゆる計算処理作業に適用されます。

## rosettaユーザーにとっての意味

### 標準的な言語（フランス語、日本語、スペイン語など）の場合

通常通りrosettaを使用してください。これらの言語には、公開されている大規模なコーパスや確立された翻訳APIがあり、主権に関する懸念はありません。必要に応じて翻訳、同期、公開を行ってください。

### 先住民言語や低リソース言語の場合

状況は根本的に異なります。

1. **まず同意を得る。** 先住民言語の翻訳メソッドを構築する前に、コミュニティとの関係を確立してください。コミュニティの関与なしに構築されたメソッドは、技術的にどれほど優れていても、公開や配布を行うべきではありません。

2. **`api` メソッドを使用する。** 翻訳パイプラインは、コミュニティが管理するインフラストラクチャ上でホストしてください。rosettaの `api` メソッドは、このために設計されています。メソッドを機能させるプロンプト、辞書、またはコーチングデータを公開することなく、キーを送信して翻訳を受け取ります。

    ```json title="Community-controlled setup"
    {
      "pairs": {
        "en:crk": {
          "method": "api",
          "endpoint": "https://api.community-server.example/translate"
        }
      }
    }
    ```

3. **すべてを文書化する。** プラグインマニフェストの `provenance` フィールドを使用して、すべてのリソース、そのライセンス、およびコミュニティの同意を得て提供されたかどうかを記載してください。

4. **スコアはライセンスではない。** リーダーボードでの高スコアは、メソッドが技術的にうまく機能することを証明するものです。翻訳の公開、プラグインの配布、またはメソッドの商業化を許可するものではありません。決定権はコミュニティにあります。

5. **データではなくメソッドを共有する。** うまく機能する技術（例：「コーチングされたプロンプトを備えたFSTゲートLLM」など）を開発した場合は、リーダーボードでその*アーキテクチャ*と*アプローチ*を共有してください。コミュニティは、特定の言語でそれを機能させるための言語データに対する管理権を保持します。

## `api` メソッドと主権

`api` [翻訳メソッド](/docs/guides/translation-methods)は、データ主権をサポートするために特別に存在しています。その理由は以下の通りです。

| 側面 | 他のメソッド | `api` メソッド |
|--------|--------------|-------------|
| **プロンプトの保存場所** | rosettaの設定ファイル内（すべての開発者が閲覧可能） | コミュニティのサーバー上（非公開） |
| **コーチングデータの保存場所** | `.rosetta/coaching/` ディレクトリ内（gitにコミットされる） | コミュニティのサーバー上（非公開） |
| **辞書の保存場所** | プラグインディレクトリ内（プラグインとともに配布される） | コミュニティのサーバー上（非公開） |
| **パイプラインの管理者** | `rosetta sync` を実行する人 | APIを運用するコミュニティ |
| **rosettaから見えるもの** | すべて | 入力されるキーと出力される翻訳 |

`api` メソッドは、意図的なアーキテクチャ上の選択です。言語の知識、文法規則、慎重に厳選されたコーチングの例などの知的財産（IP）は、ツールではなくコミュニティに属しているため、これは「ダムパイプ」として機能します。

実装の詳細については、[API経由でメソッドを提供する](/docs/guides/serving-a-method)を参照してください。
