# 数据主权

针对原住民语言的机器翻译引发了一些在法语或日语中不存在的问题。谁拥有训练数据？谁控制语言模型的表达方式？谁来决定翻译质量是否足以发布？

**答案永远是社区。**

rosetta 的构建正是为了支持这一点。`api` 方法将所有语言资源保留在服务器端，由社区控制。插件系统将方法与工具分离开来。但工具无法强制执行道德规范——本页解释了你应该遵循的原则。

---

## OCAP® 原则

**OCAP**（所有权、控制权、访问权、占有权）是由[第一民族信息治理中心](https://fnigc.ca/ocap-training/)（FNIGC）制定的一套原则，确立了应该如何收集、保护、使用和共享第一民族的数据。

| 原则 | 对翻译的意义 |
|-----------|------------------------------|
| **所有权** | 社区拥有其语言数据——词典、语法、平行文本、辅导文件，以及由此产生的任何翻译。 |
| **控制权** | 社区控制其语言数据的使用方式、谁有访问权限，以及哪些翻译方法是可接受的。 |
| **访问权** | 无论语言资源存储在哪里，社区成员都有权访问和管理自己的语言资源。 |
| **占有权** | 物理数据（辅导文件、词典、模型权重）必须存放在社区控制的基础设施上，而不是第三方云端。 |

### OCAP 在实践中的意义

- 未经社区明确授权，**不要发布**原住民语言的翻译。
- 没有数据共享协议，**不要使用**社区提供的语言数据训练模型。
- **不要从**网站、社交媒体或教育材料中抓取社区语言资源。
- **使用 `api` 方法**，以便提示词、辅导数据和词典保留在社区控制的服务器上。rosetta 的 `api` 方法是一个“哑管道”——它发送键值并获取翻译结果。所有语言知识产权（IP）都保留在服务器端。
- **记录数据来源**——[插件清单](/docs/reference/plugin-spec)中的 `provenance` 字段应列出使用的每个资源、其许可证和来源。

:::warning OCAP® 是注册商标
OCAP® 是第一民族信息治理中心的注册商标。它专门适用于加拿大的第一民族。这些原则具有更广泛的现实意义，但商标和治理权属于 FNIGC。
:::

## CARE 原则

**原住民数据治理 CARE 原则**由[全球原住民数据联盟](https://www.gida-global.org/care)（GIDA）制定，作为对 FAIR 数据原则的补充。FAIR 认为数据应该是可发现的（Findable）、可访问的（Accessible）、可互操作的（Interoperable）和可重用的（Reusable）。CARE 认为这还不够——数据治理还必须以原住民权利为中心。

| 原则 | 应用 |
|-----------|------------|
| **集体利益** | 翻译工具应首先造福语言社区。排行榜分数是改进方法的手段，而不是从社区语言中榨取商业价值。 |
| **控制权** | 社区有权管理其语言数据的收集、使用和共享方式。排行榜高分并不意味着获得了发布翻译的许可。 |
| **责任** | 处理原住民语言数据的研究人员和开发者有责任建立联系、获取同意并分享利益。 |
| **道德** | 必须将原住民的权利和福祉放在首位。翻译方法的开发应该是*与*社区合作，而不是*针对*他们。 |

## Te Mana Raraunga — 毛利数据主权

**Te Mana Raraunga** 是[毛利数据主权网络](https://www.temanararaunga.maori.nz/)。该网络主张毛利数据（包括语言数据）是一种 taonga（珍宝），受《怀唐伊条约》原则和 tikanga Māori（毛利习惯法）的约束。

核心原则：

| 原则 | 含义 |
|-----------|---------|
| **Rangatiratanga**（权威） | 毛利人拥有对其数据（包括语言数据）行使权威的固有权利。 |
| **Whakapapa**（联系） | 数据具有起源和联系。语言数据承载着创造它的人们的联系和知识。 |
| **Whanaungatanga**（义务） | 持有或处理毛利数据的人对数据来源社区负有对等义务。 |
| **Kotahitanga**（集体利益） | 毛利数据应为毛利人的集体利益而使用。 |
| **Manaakitanga**（互惠） | 毛利数据的使用应体现关怀、尊重和互惠。 |
| **Kaitiakitanga**（监护） | 数据监护人有责任保护数据并确保其被合理使用。 |

这些原则适用于 te reo Māori（毛利语）以及任何涉及毛利语数据的计算工作。

## 这对 rosetta 用户意味着什么

### 对于标准语言（法语、日语、西班牙语...）

正常使用 rosetta。这些语言拥有大量公开可用的语料库、成熟的翻译 API，且不存在主权问题。你可以随意翻译、同步和发布。

### 对于原住民语言和低资源语言

情况则截然不同：

1. **首先获得同意。** 在为原住民语言构建翻译方法之前，请先与社区建立联系。没有社区参与而构建的方法——无论技术上多么令人惊叹——都不应发布或分发。

2. **使用 `api` 方法。** 将翻译管道托管在社区控制的基础设施上。rosetta 中的 `api` 方法专为此设计：它发送键值并获取翻译结果，而不会暴露使该方法生效的提示词、词典或辅导数据。

    ```json title="Community-controlled setup"
    {
      "pairs": {
        "en:crk": {
          "method": "api",
          "endpoint": "https://api.community-server.example/translate"
        }
      }
    }
    ```

3. **记录一切。** 使用插件清单中的 `provenance` 字段列出每个资源、其许可证以及是否在社区同意下提供。

4. **分数不是许可证。** 排行榜上的高分证明该方法在技术上运行良好。它并不授予发布翻译、分发插件或将该方法商业化的许可。决定权在社区。

5. **分享方法，而不是数据。** 如果你开发了一种效果很好的技术（例如，“带有辅导提示词的 FST 门控 LLM”），请在排行榜上分享*架构*和*方法*。社区保留对使其特定语言生效的语言数据的控制权。

## `api` 方法与主权

`api` [翻译方法](/docs/guides/translation-methods)专门用于支持数据主权。原因如下：

| 方面 | 其他方法 | `api` 方法 |
|--------|--------------|-------------|
| **提示词存放位置** | 在 rosetta 的配置文件中（对所有开发者可见） | 在社区的服务器上（私有） |
| **辅导数据存放位置** | 在 `.rosetta/coaching/` 目录中（提交到 git） | 在社区的服务器上（私有） |
| **词典存放位置** | 在插件目录中（随插件分发） | 在社区的服务器上（私有） |
| **谁控制管道** | 运行 `rosetta sync` 的任何人 | 运营 API 的社区 |
| **rosetta 能看到什么** | 一切内容 | 输入键值，输出翻译 |

`api` 方法是经过深思熟虑的架构选择。它是一个“哑管道”，因为知识产权（IP）——语言知识、语法规则、精心策划的辅导示例——属于社区，而不是工具。

有关实现细节，请参阅[通过 API 提供方法](/docs/guides/serving-a-method)。
