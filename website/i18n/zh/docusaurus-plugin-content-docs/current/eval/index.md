# MT 评估

rosetta 包含一个机器翻译评估框架，专为翻译方法的**可重复基准测试**而设计——特别是对于缺乏标准 MT 基准且难以验证质量声明的低资源和原住民语言。

---

## 排行榜

核心是**[方法排行榜](/leaderboard)**——一个由 Supabase 支持的实时记分板，研究人员和社区成员可以在这里提交并比较翻译方法，这些方法都经过了带有指纹的、可重复的评估。

每次提交都包含：

- **带有指纹的流水线**——与特定的 Git 提交和配置哈希绑定，因此结果可以追溯到生成它们的具体代码
- **版本化数据集**——经过内容哈希处理和版本控制；分数只能在相同的数据集版本内进行比较
- **标准化指标**——所有评分均由共享的评估工具计算，消除了实现上的差异
- **信任层级**——自我基准测试 (Self-benchmarked)、GDS 验证 (GDS Verified) 或社区验证 (Community Validated)
- **成本跟踪**——每次提交的 API 成本，使成本与质量的权衡保持透明

排行榜目前跟踪三个指标：

| 指标 | 类型 | 测量内容 |
|--------|------|------------------|
| **chrF++** | 字符 n-gram F-score | 主要质量指标——与人类判断高度相关，特别是对于形态丰富的语言 |
| **Exact Match** | 完全匹配比例 | 严格准确率——翻译结果与黄金标准完全一致的频率是多少？ |
| **FST Acceptance** | 形态学门控通过率 | 针对带有有限状态转换器 (finite-state transducer) 验证的方法——输出中形态学有效的比例是多少？ |

## 可用数据集

### EDTeKLA Development Set v1

首个评估数据集，专为英语→平原克里语 (Plains Cree, SRO) 翻译而构建。由阿尔伯塔大学的 [EdTeKLA 研究组](https://spaces.facsci.ualberta.ca/edtekla/)创建。

| 属性 | 值 |
|----------|-------|
| **ID** | `edtekla-dev-v1` |
| **语言对** | EN → CRK（平原克里语，SRO 正字法） |
| **条目数量** | 124 |
| **许可证** | [CC BY-NC-SA 4.0](https://creativecommons.org/licenses/by-nc-sa/4.0/) |
| **出处** | `gold_standard`（由母语者验证），`textbook`（已出版的教育材料） |

### FLORES+ Devtest

由 [Open Language Data Initiative (OLDI)](https://huggingface.co/datasets/openlanguagedata/flores_plus) 维护的广泛覆盖的多语言基准测试。

| 属性 | 值 |
|----------|-------|
| **语言对** | EN → 39 种语言（所有 rosetta 注册语言） |
| **条目数量** | 每种语言 1,012 个句子 |
| **许可证** | [CC BY-SA 4.0](https://creativecommons.org/licenses/by-sa/4.0/) |
| **来源** | 最初为 Meta FLORES-200，现由 OLDI 维护 |
| **位置** | 预提取的测试数据位于主 rosetta 仓库的 `test/benchmark/fixtures/` |

有关完整的数据集模式、难度层级以及如何创建你自己的数据集，请参阅[评估数据集](/docs/eval/datasets)。

:::danger 请勿在评估数据上进行训练

**这些数据集仅用于评估。** 任何经过训练、微调、少样本提示 (few-shot-prompted) 或以其他方式接触过评估数据的方法，都会产生人为虚高的分数，并被**取消排行榜资格。**

这不是建议——这是评估完整性最重要的一条规则。请使用独立的语料库进行训练。在开发过程中，你的模型绝不能接触评估集。

如果你正在使用辅导数据或少样本示例，它们必须来自**完全独立的来源**。如有疑问，请勿包含。
:::

:::warning LLM 的非确定性

LLM 的输出具有非确定性。分数代表特定模型版本和 API 配置下的时间点测量结果。模型提供商可能随时更新权重、解码策略或安全过滤器，这会导致不同运行之间的分数发生偏移。排行榜会记录每次提交的确切模型标识符 (slug) 和时间戳。
:::

## 什么是好的方法

并非所有方法都生而平等。以下是区分严谨工作与虚高分数的关键。

### 强大方法的特征

- **训练和评估数据的清晰分离**——你的方法在开发、调优、提示词工程或少样本示例选择期间从未接触过评估集
- **可重复**——其他人可以克隆你的仓库，运行评估工具，并获得相同的分数（在 LLM 非确定性范围内）
- **文档完善**——你的[方法卡片](/docs/eval/methods)描述了你的方法的作用、使用的工具及其局限性
- **诚实说明适用范围**——如果你的方法仅适用于一种语言对，请说明；如果在某些形态模式下性能下降，请记录下来
- **社区意识**——对于原住民语言，你的方法尊重数据主权。你已咨询过语言社区，或仅使用开放许可的数据

### 危险信号（会导致取消资格的情况）

| 危险信号 | 为什么这是一个问题 |
|----------|--------------------|
| 在评估数据上训练 | 完全违背了评估的目的。虚高的分数会误导所有人。 |
| 挑选有利结果 (Cherry-picking) | 运行 10 次并提交最好的一次，而不披露其他结果 |
| 未披露的后处理 | 在评分前手动修复输出 |
| 受污染的辅导数据 | 将评估集示例用作少样本提示或字典条目 |
| 在没有出处的情况下声称具备商业可用性 | 如果你的方法使用了 CC BY-NC-SA 数据，它就不具备商业可用性 |

### 排行榜中的质量层级

排行榜支持三个信任层级：

| 层级 | 含义 | 如何获取 |
|------|---------|---------------|
| **自我基准测试 (Self-benchmarked)** | 你自己运行了评估工具并提交了结果 | 提交包含你的运行卡片 (run card) 的 PR |
| **GDS 验证 (GDS Verified)** | rosetta 维护者重现了你的结果 | 将你的方法作为可安装插件提交 |
| **社区验证 (Community Validated)** | 独立的社区成员重现了结果 | 敬请期待 |

## 如何提交

1. **构建你的方法**——有关方法接口，请参阅[构建方法](/docs/eval/methods)
2. **运行评估工具**——有关设置和用法，请参阅[评估工具](/docs/eval/harness)
3. **生成运行卡片**——评估工具会生成一个包含你的分数、指纹和元数据的 JSON 运行卡片
4. **提交 PR**——将你的运行卡片提交到 [eval harness 仓库](https://github.com/gamedaysuits/gds-mt-eval-harness)
5. **出现在排行榜上**——合并后，你的结果将出现在[方法排行榜](/leaderboard)上

## 未来方向

- **FLORES+ 模型比较运行**——对所有 39 种 rosetta 语言的前沿模型（GPT-5.5、Claude Opus 4.7、Gemini 3.1 Pro 等）进行系统评估
- **更多语言对**——随着社区验证的数据集可用，将支持克丘亚语 (Quechua)、因纽特语 (Inuktitut) 及其他低资源语言
- **数据集导入**——将外部评估数据集（WMT、Tatoeba 等）转换为 rosetta 评估格式的工具
- **自动重新运行**——检测模型版本变更并重新运行基准测试，以跟踪分数偏移
