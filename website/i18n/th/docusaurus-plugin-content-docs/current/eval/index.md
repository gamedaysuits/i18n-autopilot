# การประเมิน MT

rosetta ประกอบด้วยเฟรมเวิร์กการประเมิน machine translation ที่ออกแบบมาเพื่อ **reproducible benchmarking** (การวัดประสิทธิภาพที่สามารถทำซ้ำได้) ของวิธีการแปล — โดยเฉพาะสำหรับภาษาที่มีทรัพยากรน้อยและภาษาชนพื้นเมืองที่ไม่มีเกณฑ์มาตรฐาน MT ทั่วไป และการกล่าวอ้างด้านคุณภาพนั้นตรวจสอบได้ยาก

---

## Leaderboard

ส่วนสำคัญคือ **[Method Leaderboard](/leaderboard)** — กระดานคะแนนแบบเรียลไทม์ที่รองรับโดย Supabase ซึ่งนักวิจัยและสมาชิกในชุมชนสามารถส่งและเปรียบเทียบวิธีการแปลด้วยการประเมินที่มีการทำ fingerprint และสามารถทำซ้ำได้

ทุกการส่งข้อมูลจะประกอบด้วย:

- **Fingerprinted pipeline** — ผูกกับ Git commit และ config hash เฉพาะ เพื่อให้สามารถตรวจสอบผลลัพธ์ย้อนกลับไปยังโค้ดที่สร้างผลลัพธ์นั้นได้อย่างแม่นยำ
- **Versioned dataset** — มีการทำ content-hash และกำหนดเวอร์ชัน; คะแนนจะสามารถเปรียบเทียบกันได้เฉพาะในชุดข้อมูลเวอร์ชันเดียวกันเท่านั้น
- **Standardised metrics** — การให้คะแนนทั้งหมดถูกคำนวณโดย evaluation harness ที่ใช้ร่วมกัน เพื่อขจัดความแตกต่างในการนำไปใช้งาน (implementation)
- **Trust tiers** — ระดับความน่าเชื่อถือ ได้แก่ self-benchmarked, GDS Verified หรือ Community Validated
- **Cost tracking** — ต้นทุน API ต่อการส่งข้อมูล เพื่อให้เห็นความคุ้มค่าระหว่างต้นทุนและคุณภาพอย่างโปร่งใส

ปัจจุบัน Leaderboard ติดตามตัวชี้วัด 3 รายการ:

| ตัวชี้วัด | ประเภท | สิ่งที่วัด |
|--------|------|------------------|
| **chrF++** | Character n-gram F-score | ตัวชี้วัดคุณภาพหลัก — มีความสอดคล้องกับการตัดสินของมนุษย์ได้ดี โดยเฉพาะสำหรับภาษาที่มีความซับซ้อนทางสัณฐานวิทยา |
| **Exact Match** | สัดส่วนของการจับคู่ที่สมบูรณ์ | ความแม่นยำแบบเข้มงวด — การแปลตรงกับมาตรฐานอ้างอิง (gold standard) พอดีบ่อยแค่ไหน? |
| **FST Acceptance** | อัตราการผ่านเกณฑ์ทางสัณฐานวิทยา | สำหรับวิธีที่มีการตรวจสอบด้วย finite-state transducer — สัดส่วนของผลลัพธ์ที่ถูกต้องตามหลักสัณฐานวิทยามีเท่าใด? |

ดู [ชุดข้อมูลการประเมิน](/docs/eval/datasets) สำหรับสคีมาของชุดข้อมูลแบบเต็ม ระดับความยาก และวิธีสร้างชุดข้อมูลของคุณเอง

:::danger ห้ามฝึกสอน (TRAIN) ด้วยข้อมูลการประเมิน

**ชุดข้อมูลเหล่านี้มีไว้สำหรับการประเมินเท่านั้น** วิธีการที่ถูกฝึกสอน (trained), ปรับแต่ง (fine-tuned), ใช้ few-shot prompt หรือสัมผัสกับข้อมูลการประเมินในทางใดทางหนึ่ง จะทำให้ได้คะแนนที่สูงเกินจริงและจะถูก **ตัดสิทธิ์จาก Leaderboard**

นี่ไม่ใช่คำแนะนำ — แต่เป็นกฎที่สำคัญที่สุดของความสมบูรณ์ในการประเมิน คุณต้องใช้คลังข้อมูล (corpora) แยกต่างหากสำหรับการฝึกสอน ชุดข้อมูลการประเมินจะต้องไม่ถูกมองเห็นโดยโมเดลของคุณในระหว่างการพัฒนา

หากคุณกำลังใช้ข้อมูลการสอน (coaching data) หรือตัวอย่าง few-shot ข้อมูลเหล่านั้นต้องมาจาก **แหล่งที่มาที่แยกต่างหากโดยสิ้นเชิง** หากไม่แน่ใจ ไม่ควรนำมารวมไว้
:::

:::warning ความไม่แน่นอนของ LLM (LLM non-determinism)

ผลลัพธ์ของ LLM มีความไม่แน่นอน (non-deterministic) คะแนนจะแสดงถึงการวัดผล ณ จุดเวลาหนึ่งภายใต้เวอร์ชันของโมเดลและการกำหนดค่า API ที่เฉพาะเจาะจง ผู้ให้บริการโมเดลอาจอัปเดตน้ำหนัก (weights), กลยุทธ์การถอดรหัส (decoding strategies) หรือตัวกรองความปลอดภัยเมื่อใดก็ได้ ซึ่งอาจทำให้คะแนนคลาดเคลื่อนระหว่างการรันแต่ละครั้ง Leaderboard จะบันทึก model slug และประทับเวลา (timestamp) ที่แน่นอนสำหรับการส่งข้อมูลทุกครั้ง
:::

## สิ่งที่ทำให้วิธีการแปลมีคุณภาพ

ไม่ใช่วิธีการทั้งหมดจะถูกสร้างมาเท่าเทียมกัน นี่คือสิ่งที่แยกความแตกต่างระหว่างงานที่ทำอย่างเข้มงวดกับคะแนนที่สูงเกินจริง

### ลักษณะของวิธีการที่แข็งแกร่ง

- **Clean separation of train and eval data** — การแยกข้อมูลฝึกสอนและข้อมูลประเมินอย่างชัดเจน วิธีการของคุณต้องไม่เคยเห็นชุดข้อมูลการประเมินในระหว่างการพัฒนา การปรับแต่ง การทำ prompt engineering หรือการเลือกตัวอย่าง few-shot
- **Reproducible** — สามารถทำซ้ำได้ บุคคลอื่นสามารถโคลน repo ของคุณ รัน harness และได้คะแนนเท่ากัน (ภายใต้ขอบเขตความไม่แน่นอนของ LLM)
- **Documented** — มีการจัดทำเอกสาร [method card](/docs/eval/methods) ของคุณอธิบายว่าวิธีการของคุณทำอะไร ใช้เครื่องมืออะไรบ้าง และมีข้อจำกัดอะไรบ้าง
- **Honest about scope** — ซื่อสัตย์เกี่ยวกับขอบเขต หากวิธีการของคุณใช้ได้กับคู่ภาษาเดียวเท่านั้น ให้ระบุตามนั้น หากประสิทธิภาพลดลงในรูปแบบทางสัณฐานวิทยาบางอย่าง ให้บันทึกไว้ในเอกสาร
- **Community-aware** — ตระหนักถึงชุมชน สำหรับภาษาชนพื้นเมือง วิธีการของคุณเคารพอำนาจอธิปไตยของข้อมูล คุณได้ปรึกษากับชุมชนเจ้าของภาษาหรือใช้เฉพาะข้อมูลที่มีใบอนุญาตแบบเปิดเท่านั้น

### สัญญาณเตือน (สิ่งที่จะทำให้ถูกตัดสิทธิ์)

| สัญญาณเตือน | ทำไมจึงเป็นปัญหา |
|----------|--------------------|
| การฝึกสอนด้วยข้อมูลประเมิน | ทำลายจุดประสงค์ของการประเมินโดยสิ้นเชิง คะแนนที่สูงเกินจริงจะทำให้ทุกคนเข้าใจผิด |
| การเลือกเฉพาะผลลัพธ์ที่ดี (Cherry-picking) | รัน 10 ครั้งและส่งผลลัพธ์ที่ดีที่สุดโดยไม่เปิดเผยผลลัพธ์อื่นๆ |
| การประมวลผลหลังการแปลที่ไม่เปิดเผย | การแก้ไขผลลัพธ์ด้วยตนเองก่อนการให้คะแนน |
| ข้อมูลการสอนที่ปนเปื้อน | การใช้ตัวอย่างจากชุดข้อมูลประเมินเป็น few-shot prompt หรือรายการพจนานุกรม |
| การอ้างความพร้อมเชิงพาณิชย์โดยไม่มีแหล่งที่มา | หากวิธีการของคุณใช้ข้อมูล CC BY-NC-SA จะถือว่ายังไม่พร้อมสำหรับเชิงพาณิชย์ |

### ระดับคุณภาพใน Leaderboard

Leaderboard รองรับระดับความน่าเชื่อถือ 3 ระดับ:

| ระดับ | ความหมาย | วิธีการได้รับ |
|------|---------|---------------|
| **Self-benchmarked** | คุณรัน harness ด้วยตัวเองและส่งผลลัพธ์ | เปิด PR พร้อมกับ run card ของคุณ |
| **GDS Verified** | ผู้ดูแล rosetta สามารถทำซ้ำผลลัพธ์ของคุณได้ | ส่งวิธีการของคุณเป็นปลั๊กอินที่สามารถติดตั้งได้ |
| **Community Validated** | สมาชิกในชุมชนอิสระสามารถทำซ้ำผลลัพธ์ได้ | เร็วๆ นี้ |

## วิธีการส่งข้อมูล

1. **สร้างวิธีการของคุณ** — ดู [การสร้างวิธีการ](/docs/eval/methods) สำหรับอินเทอร์เฟซของวิธีการ
2. **รัน harness** — ดู [Eval Harness](/docs/eval/harness) สำหรับการตั้งค่าและการใช้งาน
3. **สร้าง run card** — harness จะสร้าง run card รูปแบบ JSON ที่มีคะแนน, fingerprint และข้อมูลเมตาของคุณ
4. **เปิด PR** — ส่ง run card ของคุณไปยัง [eval harness repository](https://github.com/gamedaysuits/gds-mt-eval-harness)
5. **ปรากฏบน Leaderboard** — เมื่อถูก merge แล้ว ผลลัพธ์ของคุณจะปรากฏบน [Method Leaderboard](/leaderboard)

## ทิศทางในอนาคต

- **FLORES+ model comparison runs** — การประเมินอย่างเป็นระบบของโมเดลระดับแนวหน้า (GPT-5.5, Claude Opus 4.7, Gemini 3.1 Pro ฯลฯ) ใน 39 ภาษาของ rosetta ทั้งหมด
- **More language pairs** — คู่ภาษาเพิ่มเติม เช่น Quechua, Inuktitut และภาษาที่มีทรัพยากรน้อยอื่นๆ เมื่อมีชุดข้อมูลที่ได้รับการตรวจสอบจากชุมชนพร้อมใช้งาน
- **Dataset import** — เครื่องมือสำหรับแปลงชุดข้อมูลการประเมินภายนอก (WMT, Tatoeba ฯลฯ) ให้อยู่ในรูปแบบการประเมินของ rosetta
- **Automated re-runs** — การตรวจจับการเปลี่ยนแปลงเวอร์ชันของโมเดลและรันเกณฑ์มาตรฐานซ้ำโดยอัตโนมัติเพื่อติดตามความคลาดเคลื่อนของคะแนน
