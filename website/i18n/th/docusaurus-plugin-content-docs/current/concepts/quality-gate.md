# Quality Gate

ทุกการแปลจะผ่าน validation gate ที่ทำงานแบบ deterministic ก่อนที่จะถูกเขียนลงดิสก์ครับ Quality Gate จะช่วยดักจับรูปแบบความล้มเหลวที่พบบ่อยของ machine translation — จะไม่มีการ fallback แบบเงียบ ๆ และไม่มีข้อมูลขยะถูกเขียนลงในไฟล์ locale ของคุณครับ

## การตรวจสอบความถูกต้อง

| การตรวจสอบ | สิ่งที่ดักจับได้ | Gate Label |
|-------|----------------|-----------|
| **Empty/blank** | โมเดลส่งคืนค่าเป็น empty string หรือ whitespace | `[GATE] empty` |
| **Source echo** | โมเดลส่งคืนค่าเป็นข้อความภาษาอังกฤษต้นฉบับ | `[GATE] source-echo` |
| **Hallucination loop** | รูปแบบ trigram ที่ซ้ำกัน (เช่น `"Qo' Qo' Qo'"`) | `[GATE] hallucination` |
| **Length inflation** | ผลลัพธ์มีความยาวมากกว่าต้นฉบับอย่างมีนัยสำคัญ | `[GATE] length` |
| **Script compliance** | สคริปต์ (script) ไม่ถูกต้องสำหรับ locale ปลายทาง | `[GATE] script` |

### Empty/Blank

ปฏิเสธการแปลที่เป็น empty string, มีเฉพาะ whitespace หรือ `null` ครับ การตรวจสอบนี้จะดักจับโมเดลที่ไม่ส่งคืนค่าใด ๆ สำหรับคีย์ (keys) ที่ยากครับ

### Source Echo

ตรวจจับเมื่อโมเดลส่งคืนข้อความต้นฉบับภาษาอังกฤษแทนที่จะแปลครับ ซึ่งมักพบได้บ่อยในข้อความสั้น ๆ และ prompt ที่ระบุรายละเอียดไม่เพียงพอครับ

### Hallucination Loop

วิเคราะห์รูปแบบ trigram (3 ตัวอักษร) ในผลลัพธ์ครับ หากมี trigram ใดซ้ำกันมากกว่าจำนวน threshold ที่กำหนดไว้เมื่อเทียบกับความยาวของผลลัพธ์ การแปลนั้นจะถูกปฏิเสธครับ การตรวจสอบนี้จะดักจับผลลัพธ์ที่เสื่อมสภาพ (degenerate outputs) เช่น `"Qo' Qo' Qo' Qo' Qo'"` ครับ

### Length Inflation

ปฏิเสธการแปลที่ความยาวของผลลัพธ์เกิน `maxLengthRatio × source length` (ค่าเริ่มต้น: 4 เท่า) ครับ การตรวจสอบนี้จะดักจับอาการ hallucination ของโมเดลที่สร้างข้อความยาวเหยียดสำหรับอินพุตสั้น ๆ ครับ

สามารถกำหนดค่าได้ผ่าน `maxLengthRatio` ใน config ของคุณครับ

### Script Compliance

สำหรับ locale ที่มีการกำหนดค่าฟิลด์ `script` (เช่น `"script": "cans"` สำหรับ Plains Cree Syllabics) จะตรวจสอบว่าผลลัพธ์มีตัวอักษร non-ASCII ที่เหมาะสมกับสคริปต์เป้าหมายหรือไม่ครับ ผลลัพธ์ที่มีเฉพาะอักษรละติน (Latin-only) สำหรับ locale ภาษาอาหรับ, CJK หรือ Syllabics จะถูกปฏิเสธครับ

## สิ่งที่เกิดขึ้นเมื่อเกิดความล้มเหลว

1. การแปลที่ล้มเหลวจะถูกบันทึกลงใน stderr พร้อมกับคำนำหน้า `[GATE]`, ชื่อคีย์, สาเหตุ และตัวอย่างของค่า (value) ครับ
2. คีย์ดังกล่าวจะ **ไม่** ถูกเขียนลงในไฟล์ locale ครับ
3. กระบวนการ retry cascade จะเริ่มทำงาน (ดูด้านล่าง) ครับ

```
[GATE] hero.title: source-echo — "Welcome to our platform"
[GATE] nav.about: hallucination — "À À À À À À À À"
```

## Retry Cascade

เมื่อ batch ล้มเหลว (เกิดข้อผิดพลาดในการแยกวิเคราะห์ JSON หรือถูกปฏิเสธโดย Quality Gate) rosetta จะลองใหม่ (retry) ด้วยขนาด batch ที่เล็กลงเรื่อย ๆ ครับ:

```
Full batch (30 keys) → parse error
  └→ Half batch (15 keys) → 2 failures
      └→ Individual keys (1 each) → isolates the 2 problem keys
```

งบประมาณในการลองใหม่ (retry budget) จะถูกจำกัดโดย `maxRetries` (ค่าเริ่มต้น: 3, สามารถกำหนดค่าได้ในแต่ละภาษา) ครับ สิ่งนี้ช่วยป้องกันการใช้ token อย่างควบคุมไม่ได้กับคีย์ที่ล้มเหลวอย่างต่อเนื่องครับ

หลังจากใช้สิทธิ์การลองใหม่จนหมด คีย์ที่มีปัญหาจะถูกบันทึกและข้ามไปครับ โดยจะถูกนำมาลองใหม่อีกครั้งในการรัน `sync` ครั้งถัดไปครับ

## Prompt Caching

ข้อความระบบ (system message) (ระดับภาษา, กฎไวยากรณ์, บันทึกรูปแบบ) จะถูกแยกออกจากข้อความผู้ใช้ (user message) (คีย์ที่ต้องการแปล) ครับ การแยกส่วนนี้เป็นความตั้งใจ:

- ข้อความระบบจะ **เหมือนกันในทุก ๆ batch** สำหรับ locale ที่กำหนดครับ
- ผู้ให้บริการอย่าง Anthropic และ Google จะทำการแคช (cache) ข้อความระบบที่ซ้ำกันครับ
- ผลลัพธ์: batch แรกจะจ่ายค่า token เต็มจำนวน ส่วน batch ถัด ๆ ไปจะจ่ายเฉพาะส่วนของข้อความผู้ใช้เท่านั้นครับ

สิ่งนี้สามารถลดต้นทุนค่า token ได้อย่างมากสำหรับโปรเจกต์ที่มีหลาย batch ครับ
