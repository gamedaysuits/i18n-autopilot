# อธิปไตยของข้อมูล

การแปลภาษาด้วยเครื่อง (Machine translation) สำหรับภาษาพื้นเมือง (Indigenous languages) ทำให้เกิดคำถามที่ไม่มีอยู่ในภาษาฝรั่งเศสหรือภาษาญี่ปุ่น ใครเป็นเจ้าของข้อมูลการฝึกอบรม (training data)? ใครเป็นผู้ควบคุมวิธีการสื่อสารของโมเดลภาษา? ใครเป็นผู้ตัดสินว่าคำแปลนั้นดีพอที่จะเผยแพร่หรือไม่?

**คำตอบคือชุมชนเสมอ**

rosetta ถูกสร้างขึ้นมาเพื่อรองรับสิ่งนี้ เมธอด `api` จะเก็บทรัพยากรทางภาษาทั้งหมดไว้ที่ฝั่งเซิร์ฟเวอร์ภายใต้การควบคุมของชุมชน ระบบปลั๊กอินจะแยกเมธอดออกจากเครื่องมือ แต่เครื่องมือไม่สามารถบังคับใช้จริยธรรมได้ — หน้านี้จะอธิบายถึงหลักการที่คุณควรปฏิบัติตาม

---

## หลักการ OCAP®

**OCAP** (Ownership, Control, Access, Possession) คือชุดหลักการที่พัฒนาโดย [First Nations Information Governance Centre](https://fnigc.ca/ocap-training/) (FNIGC) ซึ่งกำหนดวิธีการรวบรวม ปกป้อง ใช้ และแบ่งปันข้อมูลของกลุ่ม First Nations

| หลักการ | ความหมายสำหรับการแปล |
|-----------|------------------------------|
| **Ownership** (ความเป็นเจ้าของ) | ชุมชนเป็นเจ้าของข้อมูลทางภาษาของตนเอง — พจนานุกรม ไวยากรณ์ ข้อความคู่ขนาน (parallel texts) ไฟล์การสอน (coaching files) และคำแปลใดๆ ที่สร้างขึ้นจากข้อมูลเหล่านั้น |
| **Control** (การควบคุม) | ชุมชนเป็นผู้ควบคุมวิธีการใช้ข้อมูลภาษาของตน ใครบ้างที่สามารถเข้าถึงได้ และเมธอดการแปลใดที่สามารถยอมรับได้ |
| **Access** (การเข้าถึง) | สมาชิกในชุมชนมีสิทธิ์เข้าถึงและจัดการทรัพยากรภาษาของตนเอง ไม่ว่าจะถูกจัดเก็บไว้ที่ใดก็ตาม |
| **Possession** (การครอบครอง) | ข้อมูลทางกายภาพ (ไฟล์การสอน พจนานุกรม ค่าน้ำหนักของโมเดล) ต้องอยู่บนโครงสร้างพื้นฐานที่ชุมชนควบคุม — ไม่ใช่บนคลาวด์ของบุคคลที่สาม |

### ความหมายของ OCAP ในทางปฏิบัติ

- **ห้ามเผยแพร่คำแปล** ของภาษาพื้นเมืองโดยไม่ได้รับอนุญาตอย่างชัดเจนจากชุมชน
- **ห้ามฝึกอบรมโมเดล** ด้วยข้อมูลทางภาษาที่ชุมชนจัดเตรียมให้โดยไม่มีข้อตกลงการแบ่งปันข้อมูล
- **ห้ามดึงข้อมูล (scrape)** ทรัพยากรภาษาของชุมชนจากเว็บไซต์ โซเชียลมีเดีย หรือสื่อการเรียนการสอน
- **ใช้เมธอด `api`** เพื่อให้พรอมต์ ข้อมูลการสอน และพจนานุกรมอยู่บนเซิร์ฟเวอร์ที่ควบคุมโดยชุมชน เมธอด `api` ของ rosetta ทำหน้าที่เป็นเพียง "ท่อส่งข้อมูล (dumb pipe)" — โดยจะส่งคีย์ออกไปและรับคำแปลกลับมา ทรัพย์สินทางปัญญา (IP) ทางภาษาทั้งหมดจะยังคงอยู่ฝั่งเซิร์ฟเวอร์
- **จัดทำเอกสารแหล่งที่มา** — ฟิลด์ `provenance` ใน [plugin manifest](/docs/reference/plugin-spec) ควรระบุทรัพยากรทุกรายการที่ใช้ สิทธิ์การใช้งาน และแหล่งที่มา

:::warning OCAP® เป็นเครื่องหมายการค้าจดทะเบียน
OCAP® เป็นเครื่องหมายการค้าจดทะเบียนของ First Nations Information Governance Centre ซึ่งบังคับใช้โดยเฉพาะกับกลุ่ม First Nations ในแคนาดา หลักการนี้มีความเกี่ยวข้องในวงกว้าง แต่เครื่องหมายการค้าและอำนาจการกำกับดูแลเป็นของ FNIGC
:::

## หลักการ CARE

**CARE Principles for Indigenous Data Governance** (หลักการ CARE สำหรับการกำกับดูแลข้อมูลพื้นเมือง) ได้รับการพัฒนาโดย [Global Indigenous Data Alliance](https://www.gida-global.org/care) (GIDA) เพื่อเป็นส่วนเสริมของหลักการข้อมูล FAIR หลักการ FAIR ระบุว่าข้อมูลควรค้นหาได้ (Findable) เข้าถึงได้ (Accessible) ทำงานร่วมกันได้ (Interoperable) และนำกลับมาใช้ใหม่ได้ (Reusable) แต่หลักการ CARE ระบุว่านั่นยังไม่เพียงพอ — การกำกับดูแลข้อมูลต้องให้ความสำคัญกับสิทธิของชนพื้นเมืองเป็นศูนย์กลางด้วย

| หลักการ | การนำไปประยุกต์ใช้ |
|-----------|------------|
| **Collective Benefit** (ผลประโยชน์ร่วมกัน) | เครื่องมือการแปลควรให้ประโยชน์แก่ชุมชนเจ้าของภาษาก่อน คะแนนบนกระดานผู้นำ (Leaderboard) เป็นเพียงวิธีการปรับปรุงเมธอด ไม่ใช่เพื่อดึงมูลค่าเชิงพาณิชย์จากภาษาของชุมชน |
| **Authority to Control** (อำนาจในการควบคุม) | ชุมชนมีอำนาจในการกำกับดูแลวิธีการรวบรวม ใช้ และแบ่งปันข้อมูลภาษาของตน คะแนนบนกระดานผู้นำที่สูงไม่ได้หมายความว่าจะได้รับอนุญาตให้เผยแพร่คำแปล |
| **Responsibility** (ความรับผิดชอบ) | นักวิจัยและนักพัฒนาที่ทำงานกับข้อมูลภาษาพื้นเมืองมีความรับผิดชอบในการสร้างความสัมพันธ์ ขอความยินยอม และแบ่งปันผลประโยชน์ |
| **Ethics** (จริยธรรม) | สิทธิและความเป็นอยู่ที่ดีของชนพื้นเมืองต้องเป็นข้อกังวลหลัก เมธอดการแปลควรได้รับการพัฒนา *ร่วมกับ* ชุมชน ไม่ใช่ทำ *เกี่ยวกับ* พวกเขา |

## Te Mana Raraunga — อธิปไตยของข้อมูลชาวเมารี

**Te Mana Raraunga** คือ [Māori Data Sovereignty Network](https://www.temanararaunga.maori.nz/) (เครือข่ายอธิปไตยของข้อมูลชาวเมารี) ซึ่งยืนยันว่าข้อมูลของชาวเมารี — รวมถึงข้อมูลทางภาษา — เป็น taonga (สมบัติ) ที่อยู่ภายใต้หลักการของสนธิสัญญาไวตางิ (Treaty of Waitangi) และ tikanga Māori (กฎหมายจารีตประเพณีของชาวเมารี)

หลักการสำคัญ:

| หลักการ | ความหมาย |
|-----------|---------|
| **Rangatiratanga** (อำนาจ) | ชาวเมารีมีสิทธิโดยกำเนิดในการใช้อำนาจเหนือข้อมูลของตน รวมถึงข้อมูลทางภาษา |
| **Whakapapa** (ความสัมพันธ์) | ข้อมูลมีต้นกำเนิดและการเชื่อมโยง ข้อมูลทางภาษานำพาความสัมพันธ์และความรู้ของผู้คนที่สร้างมันขึ้นมา |
| **Whanaungatanga** (ภาระผูกพัน) | ผู้ที่ถือครองหรือประมวลผลข้อมูลของชาวเมารีมีภาระผูกพันซึ่งกันและกันต่อชุมชนที่เป็นแหล่งที่มาของข้อมูล |
| **Kotahitanga** (ผลประโยชน์ร่วมกัน) | ข้อมูลของชาวเมารีควรถูกนำมาใช้เพื่อผลประโยชน์ร่วมกันของชาวเมารี |
| **Manaakitanga** (การแลกเปลี่ยนซึ่งกันและกัน) | การใช้ข้อมูลของชาวเมารีควรเกี่ยวข้องกับความใส่ใจ ความเคารพ และการแลกเปลี่ยนซึ่งกันและกัน |
| **Kaitiakitanga** (การพิทักษ์รักษา) | ผู้พิทักษ์ข้อมูลมีหน้าที่ปกป้องข้อมูลและตรวจสอบให้แน่ใจว่ามีการใช้อย่างเหมาะสม |

หลักการเหล่านี้บังคับใช้กับ te reo Māori (ภาษาเมารี) และงานด้านคอมพิวเตอร์ใดๆ ที่เกี่ยวข้องกับข้อมูลภาษาเมารี

## ความหมายของสิ่งนี้สำหรับผู้ใช้ rosetta

### สำหรับภาษามาตรฐาน (ฝรั่งเศส, ญี่ปุ่น, สเปน...)

คุณสามารถใช้ rosetta ได้ตามปกติ ภาษาเหล่านี้มีคลังข้อมูล (corpora) ขนาดใหญ่ที่เปิดให้ใช้งานแบบสาธารณะ มี API การแปลที่ได้รับการยอมรับ และไม่มีข้อกังวลด้านอธิปไตย คุณสามารถแปล ซิงค์ และเผยแพร่ได้ตามที่คุณต้องการ

### สำหรับภาษาพื้นเมืองและภาษาที่มีทรัพยากรน้อย (low-resource languages)

สถานการณ์จะแตกต่างออกไปอย่างสิ้นเชิง:

1. **ขอความยินยอมก่อน** ก่อนที่จะสร้างเมธอดการแปลสำหรับภาษาพื้นเมือง คุณควรสร้างความสัมพันธ์กับชุมชน เมธอดที่สร้างขึ้นโดยไม่มีการมีส่วนร่วมของชุมชน — ไม่ว่าจะน่าประทับใจในทางเทคนิคเพียงใดก็ตาม — ไม่ควรถูกเผยแพร่หรือแจกจ่าย

2. **ใช้เมธอด `api`** โฮสต์ไปป์ไลน์การแปลบนโครงสร้างพื้นฐานที่ควบคุมโดยชุมชน เมธอด `api` ใน rosetta ได้รับการออกแบบมาเพื่อสิ่งนี้: โดยจะส่งคีย์และรับคำแปลกลับมาโดยไม่เปิดเผยพรอมต์ พจนานุกรม หรือข้อมูลการสอนที่ทำให้เมธอดทำงานได้

    ```json title="Community-controlled setup"
    {
      "pairs": {
        "en:crk": {
          "method": "api",
          "endpoint": "https://api.community-server.example/translate"
        }
      }
    }
    ```

3. **จัดทำเอกสารทุกอย่าง** ใช้ฟิลด์ `provenance` ใน plugin manifest ของคุณเพื่อระบุทรัพยากรทุกรายการ สิทธิ์การใช้งาน และระบุว่าทรัพยากรนั้นได้รับความยินยอมจากชุมชนหรือไม่

4. **คะแนนไม่ใช่ใบอนุญาต** คะแนนที่สูงบนกระดานผู้นำพิสูจน์ได้ว่าเมธอดนั้นทำงานได้ดีในทางเทคนิค แต่นั่นไม่ได้เป็นการอนุญาตให้เผยแพร่คำแปล แจกจ่ายปลั๊กอิน หรือนำเมธอดไปใช้ในเชิงพาณิชย์ ชุมชนจะเป็นผู้ตัดสินใจ

5. **แบ่งปันเมธอด ไม่ใช่ข้อมูล** หากคุณพัฒนาเทคนิคที่ทำงานได้ดี (เช่น "FST-gated LLM with coached prompts") ให้แบ่งปัน *สถาปัตยกรรม* และ *แนวทาง* บนกระดานผู้นำ ชุมชนจะยังคงควบคุมข้อมูลทางภาษาที่ทำให้เทคนิคนี้ทำงานได้สำหรับภาษาเฉพาะของพวกเขา

## เมธอด `api` และอธิปไตย

[เมธอดการแปล](/docs/guides/translation-methods) `api` มีอยู่เพื่อรองรับอธิปไตยของข้อมูลโดยเฉพาะ นี่คือเหตุผล:

| แง่มุม | เมธอดอื่นๆ | เมธอด `api` |
|--------|--------------|-------------|
| **ตำแหน่งที่เก็บพรอมต์** | ในไฟล์คอนฟิกของ rosetta (นักพัฒนาทุกคนมองเห็นได้) | บนเซิร์ฟเวอร์ของชุมชน (ส่วนตัว) |
| **ตำแหน่งที่เก็บข้อมูลการสอน** | ในไดเรกทอรี `.rosetta/coaching/` (คอมมิตลงใน git) | บนเซิร์ฟเวอร์ของชุมชน (ส่วนตัว) |
| **ตำแหน่งที่เก็บพจนานุกรม** | ในไดเรกทอรีปลั๊กอิน (แจกจ่ายพร้อมกับปลั๊กอิน) | บนเซิร์ฟเวอร์ของชุมชน (ส่วนตัว) |
| **ใครเป็นผู้ควบคุมไปป์ไลน์** | ใครก็ตามที่รัน `rosetta sync` | ชุมชนที่ดำเนินการ API |
| **สิ่งที่ rosetta มองเห็น** | ทุกอย่าง | คีย์ที่ส่งเข้า, คำแปลที่ส่งออก |

เมธอด `api` เป็นทางเลือกทางสถาปัตยกรรมที่ตั้งใจไว้ มันเป็น "ท่อส่งข้อมูล (dumb pipe)" เนื่องจากทรัพย์สินทางปัญญา (IP) — ความรู้ทางภาษา กฎไวยากรณ์ ตัวอย่างการสอนที่คัดสรรมาอย่างดี — เป็นของชุมชน ไม่ใช่ของเครื่องมือ

ดู [การให้บริการเมธอดผ่าน API](/docs/guides/serving-a-method) สำหรับรายละเอียดการนำไปใช้งาน
